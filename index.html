Lesson 1: The Web & HTML 101 ("These are the breaks")

The web

"The web is a bunch of computers that communicate with each other." Computers send HTTP requests between servers. A server is land, land on which you can build your websites (houses).

HTML

Hypertext Markup Language. HTML docs form the majority of the web. In an HTML doc, the text describes "what you see" - the markup describes "how it looks". HTML documents are made of HTML elements that are differentiated by using tags. HTML elements are either inline or block (invisible boxes around the content).

The 5 ways a Programmer thinks
  • Procedural Thinking: How do you tell the dumbest thing in the world (a computer) to do what you want it to do exactly? For example, how do you tell a computer to shuffle a deck of cards? How do you tell a computer to make a PB & J sandwich?
  • Abstract Thinking: How do you find the generality (within yourself) amongst seemingly different things? (analogies, metaphors)
  • Systematic Debugging: How do you find out the problem and it's possible causes?
  • Technological Empathy: How does a computer interpret what you tell it to do? (Your code)
  • Systems Thinking: How do all these ways of thinking work together?

Lesson 2: Creating a Structured Document with HTML & CSS

Houses

A website is a house, a home - this is created with computer code. A website needs a framework, a structure - this is created by HTML. A website needs color, needs style - this is created with CSS. Windows and doors need to be opened and closed, showers and sinks need to run water - this is created with Javascript.

Files

A website (project) is comprised of HTML and CSS documents respectively. Technically, you can bake style into HTML, but it's superfluous given a given website (project) typically possesses a style consistently expressed throughout its body. For efficiency (through the practice of a separation of concerns), it's best practice to separate CSS, that can be applied to multiple HTML files of a website (project).

Boxes

Everything on a website is a rectangle. Even circles and ovals are rectangles. This is why - in designing the front-end of a website - you boxify.

Cheese

To design a website (wireframe it), you can use a computer graphics program, you can use Kraft (American Cheese) singles - ironically, most experienced front-end programmers use the good ole' pencil & paper. It's the fastest, most importantly, it's what we are most accustomed to.

Tree

DOM (Document-Object-Model) is the structure of the page (headers to divs). Mother and multiple children. Boxes inside of boxes inside of boxes.

Tips
  • You don't have to remember everything (that's what Google is for). There's too much to remember.
  • Name the div classes in your files accurately so you can identify them later.
  • All projects are iterations ("Done is better than perfect.")
  • Don't repeat yourself (DRY) [Note: One of the first principles of Programming.]
  • Use the Google Developer tools baked into the Chrome browser (you can even change elements right in the Developer tools panel).
  • There are inline elements (span) and block elements (div). Block elements have an "invisible box" around the content inside of them.
Remember
The stages of expertise:
  1. Ignorance
  2. Awareness
  3. Ability
  4. Fluency

Lesson 3: The Makings of a Great Coder (& stuff about DIV)

Live to give

A good coder writes for someone else, even for their future self. Their code is readable: proper indentation, horizontal more than vertical, 80 characters per line. "Indentation and spacing makes up 80% of your style."

80, 80

The truism goes: "When you are writing comments:
  • Beginners make comments about what they're doing
  • More experienced people make comments on how they're doing what they're doing
  • Experts make comments on why they're doing what they're doing"
Great coders:
  • Write easy to understand variables
  • Write very logical (flows into the next) structure
  • Write comments about "this is how I do it" + "this is why I do it"
  • Name their variables appropriately (ex. my_func vs count_friends)
  • Validators exist for HTML & CSS.
  • D.R.Y. - Don't Repeat Yourself. "Every piece of knowledge must have a single, unambiguous, authoritative representation within a system." - Andy Hunt/Dave Thomas
  • Avoid code bloat: Less code, less prone to code mistakes
  • If one didn't have CSS classes, one would have had to style inline which would, not only make coding more difficult because it would be more granular, but it would also make debugging difficult. One should always code for others as well as your future self who will all maintain your code.
  • If one didn't use an External style sheet for the CSS classes, one would have had to use an internal style sheet whose style would have only lived on that particular HTML file. Typically, it's reasonable to expect a given web project to consist of multiple HTML files, hence it's best practice to keep the whole project unified with similar style - similar style is guaranteed using an external style sheet. "Design changes happen in one place, rather than one hundred." - Chris Coyier
Remember
  • Always include "alt" in your img element (for visually impaired).
  • You can apply analogy to understand what is going on in the abstract ("can of beans" to understand ID vs Class)
  • Name your DIVs and spans consistently (so you can find them easier later).
  • Div takes up the width of the entire site. Span doesn't.
  • Using IDs is old school. Use "class."
